# Driver vs Car — Live Interactive Tool: Design Spec

**Date:** 2026-06-17
**Author:** Cole Richards (with Claude)
**Status:** Approved by user (pending spec review)

---

## 1. Purpose & context

Through Round 7 (Barcelona, 2026-06-14) the repo ships a **static, finished
analysis**: three notebooks → matplotlib figures → PDFs → a README, regenerated by
hand after each race via `scripts/refresh.py`. It is a strong *report* about one
driver pairing (Antonelli vs Russell).

The goal of this work is to turn that report into something **people want to use** —
specifically:

- **Primary audience: recruiters / hiring managers.** Cole is a June-2026 new grad.
  The single biggest upgrade in perceived skill is moving from "notebooks + PDFs on
  GitHub" to a **live, interactive, hosted tool with a URL**. This is also literally
  "a tool around the analysis."
- **Secondary audience: F1 fans.** A live tool is shareable (e.g. r/formula1, X). We
  design *toward* shareability without betting the project on public traction.

The reframe: from *"a static analysis of Antonelli"* to a **live "Driver vs Car" tool
covering every 2026 teammate pair**, built on the real-data Chapters 1–2 (qualifying
delta + race conversion). The Antonelli narrative + synthetic-history Chapter 3 are
**demoted to a curated case-study landing page**, not a live feature.

## 2. Why this scope (key constraints)

Three facts about the data drive every scope decision:

1. **The only car-controlled comparison is the teammate one.** "Same car → the delta
   is the driver" holds for Ch1 (qualifying delta) and Ch2 (race conversion), which use
   **real FastF1 telemetry for all 22 drivers**. Going "driver vs the field" loses the
   car control and would require a model to separate driver from car — out of scope
   (bigger statistical lift, easy to get wrong, undercuts the project's trustworthy
   brand).
2. **Chapter 3 is synthetic.** The cross-year history is, by the project's own data
   note, "not real-world F1." It is fine as a labeled case-study appendix but **cannot
   be a live, generalized feature**. It will not be generalized to arbitrary teams.
3. **Telemetry is heavy** (~900 MB cache; a full refresh takes ~10+ min). "Live"
   therefore means **pre-computed once per race weekend and published as cached JSON**,
   never computed on a user's click.

## 3. Scope

### In scope (MVP)
- A **Python pre-compute pipeline** that, for each of the 11 teams and each completed
  2026 round, computes Ch1 (qualifying) and Ch2 (race) metrics and writes JSON.
- A **React + Vite frontend** that reads the JSON and renders interactive charts:
  landing (Antonelli story + team grid), team page (the tool), method page (trust).
- A **GitHub Actions workflow** that auto-refreshes the data after each race weekend
  and redeploys to GitHub Pages, with zero manual edits.
- Honesty surfaced in the UI: sensor-freeze, small-sample, and Q-mismatch caveats.

### Out of scope (explicitly)
- Arbitrary cross-team driver-vs-driver comparison (breaks the same-car premise).
- "Driver vs the field" modeling.
- Generalizing the synthetic Chapter 3 to all teams.
- Re-creating the bespoke matplotlib overlays (track-delta map, corner-signature-on-
  track) interactively for all 11 teams — these stay as Antonelli case-study PNGs.
- Live/runtime Python; user accounts; any database.

## 4. Architecture

```
  FastF1 (cached telemetry)
        │
        ▼
 ┌─────────────────────────┐   runs once per race weekend (local OR CI)
 │  Python pipeline (src/)  │   extends existing analysis modules
 │  generalized: any pair   │ ─► writes web/public/data/*.json (small, committed)
 └─────────────────────────┘
        │  JSON artifacts (aggregates + per-lap-downsampled traces)
        ▼
 ┌─────────────────────────┐
 │  React + Vite frontend   │   reads JSON; renders interactive charts
 │  team picker + case study│   (no Python at runtime)
 └─────────────────────────┘
        │  static build (dist/)
        ▼
   GitHub Pages  ◄── GitHub Action: weekly → run pipeline → commit JSON → build → deploy
```

**Repo layout:** Python stays at the repo root; the web app lives in `web/`. The
pipeline writes to `web/public/data/`. The existing notebooks, `plotting.py`, and PDF
deliverables are **untouched** and continue to work.

## 5. Python pipeline (detail)

The existing analysis functions are **already parameterized** (`compare_teammates(…,
drv_a, drv_b)`, `compute_corner_signatures(…, drv_a, drv_b)`, `start_summary(…, drv_a,
drv_b)`, `stint_pace(…, drivers)`, `gap_to_rival(…, driver)` — ANT/RUS are defaults,
not hardwired). Generalizing is mostly feeding different driver codes in. Three new
single-purpose pieces; `plotting.py` is left as-is.

| Module | One job | Depends on | Unit-testable |
|---|---|---|---|
| `src/teams.py` (new) | Given a year, resolve the 11 teams → their driver pair from FastF1 results. Handle lineup changes. | FastF1 | yes (mock results) |
| `src/serialize.py` (new) | Pure: analysis DataFrames → JSON dicts matching a documented schema. No FastF1, no I/O. | pandas | yes, trivially |
| `scripts/build_site_data.py` (new) | Orchestrate: for each team × round, call existing analysis fns → serialize → write `web/public/data/`. Commit-if-changed friendly (deterministic output). | the above + benchmarks/race | via output schema check |

### 5.1 Pair resolution & lineup changes
Pairs are **not fixed** (reserve drivers, mid-season swaps → a team can have 3+ drivers
across a season). Plan:
- Resolve the pair **per race** from who actually drove for the team that weekend.
- Define the season's **canonical pair** as the most frequent pairing, used for the
  team page header; footnote any swap.
- The teammate-delta math only ever compares two drivers who shared the car **in the
  same race**, so this stays honest regardless of swaps.
- **Driver A/B ordering per team is fixed deterministically** (by car number) so the
  delta sign is stable across rounds and the UI can label "+ = A faster" consistently.

### 5.2 Year-over-year is conditional
The "+x s/track vs last season" comparison only works when the **same pairing drove the
same team** in the prior season. Most 2026 teams changed at least one seat, so YoY is
computed **only where the pairing carries over** (Antonelli/Mercedes will have it; many
teams won't) and the panel is **omitted otherwise** — never faked.

### 5.3 JSON artifacts
- `web/public/data/index.json` — manifest: list of teams (slug, display name, canonical
  driver pair, whether YoY is available), races covered, `lastUpdated` (UTC), data
  source/version.
- `web/public/data/teams/<slug>.json` — per team:
  - **Qualifying (Ch1):** per-round lap delta (s, + = driver A faster) + both drivers'
    Q positions + per-round caveat flags; segment-category means (straight / slow /
    medium / fast corner); corner-signature buckets (apex-speed Δ, brake-on Δ,
    throttle-full Δ with counts); YoY block (present only when available).
  - **Race (Ch2):** start conversion (grid → lap1 → finish, with DNF status), per-stint
    median pace by compound, tire-degradation slope per stint (NaN where <5 clean laps),
    a **per-lap-downsampled** gap-to-rival trace.
  - **Caveats:** sensor-freeze flags, "<5 clean laps → NaN", Q-session-mismatch flags,
    "not fuel-corrected" — all serialized so the UI can render the same warnings.
- Traces are downsampled to per-lap to keep each team file to ~tens of KB.

## 6. Frontend (detail)

**Stack:** Vite + React + TypeScript + Tailwind; charts via **Recharts** (declarative,
React-native, hover/toggle out of the box; adequate for the line/bar charts here).
Matches the existing `website` / `personal_website` / `workouts` repos. No SSR (data is
pre-built JSON).

**Screens:**
1. **Landing (`/`)** — the hook. One-line framing: *"How much of a driver's results are
   the driver, and how much is the car? Pick a team."* Leads with the **Antonelli
   case-study** (current phenomenon; reuses the existing rich figures), then a grid of
   all 11 teams.
2. **Team page (`/team/:slug`)** — the tool. The two teammates with: qualifying
   lap-delta-by-round (line), segment-category split (bars), corner-signature buckets,
   and the race-conversion panel (start, stint pace, tire deg, gap trace). Each panel
   shows **caveat badges** from the JSON. Topped by a **one-line shareable takeaway**
   (screenshot-friendly).
3. **Method (`/method`)** — trust. Reuses the README method + limitations prose.

**Data layer:** fetch `index.json` once; lazy-fetch `teams/<slug>.json` on selection.
Plain `fetch` + a small hook — no global state library needed.

**Core interaction stays focused:** team picker → team page. No arbitrary cross-team
comparisons.

## 7. Automation, testing, deployment

### 7.1 GitHub Actions workflow
- **Trigger:** weekly cron (Mondays during the season) + manual `workflow_dispatch`. A
  weekly tick that no-ops when nothing is new is simpler/robuster than per-race timing.
- **Steps:** checkout → setup Python → **restore FastF1 cache** (`actions/cache`, static
  restore-key so it always pulls latest, saves under a dated key) → run
  `build_site_data.py` (fetches only sessions not already cached; graceful per-session
  `try/except` as in `refresh.py`) → **commit `web/public/data/*.json` only if changed**
  → build the Vite app → deploy to GitHub Pages.
- A failed data step **keeps the last good deploy** rather than shipping broken output.

### 7.2 Auto-discover completed rounds (removes the last manual step)
- **The live tool pipeline auto-discovers** completed 2026 rounds from the FastF1
  schedule (EventDate ≤ today **and** race results available), so CI needs no human edit.
- **The curated Antonelli notebooks/case-study keep the manual `RACES` list** in
  `season.py`, because that prose is hand-written and updated when Cole chooses.
- This is the deliberate "tool = live, case study = curated" split.

### 7.3 Testing
- Keep the existing 16 pytest invariants.
- Add: `teams.py` pair-resolution tests (mock results → expected pairs, incl. a
  lineup-swap case); `serialize.py` schema tests (required keys; the
  segment-deltas-sum-to-lap-delta invariant survives serialization); a manifest test
  (every team has exactly 2 canonical drivers).
- Frontend: light by design — type-check + `vite build` must pass in CI before deploy;
  one Vitest render test against sample JSON.
- CI gates: `pytest` and the frontend build must pass before publishing/deploying.

### 7.4 Deployment
- **GitHub Pages** (free; matches the existing `milescoler.github.io` setup; Vite `base`
  set for the repo). Netlify/Vercel are drop-in alternatives later.

## 8. Risks & mitigations
- **CI compute time / FastF1 reliability.** Mitigated by caching + fetching only the
  newest race + graceful per-session skips + last-good-deploy on failure.
- **Telemetry for all 22 drivers is heavier than ANT/RUS-only.** Mitigated because the
  cost is incremental per race (cache persists); first full build is the only slow run.
- **Lineup changes / sparse data.** Handled by per-race pair resolution and conditional
  panels (YoY omitted when unavailable; metrics NaN-guarded on small samples).
- **Credibility.** Only real-data Ch1/Ch2 are generalized; synthetic Ch3 stays a
  labeled Antonelli case study; caveats surfaced in the UI.

## 9. Suggested milestones (for the implementation plan)
1. Generalize + pair resolution: `src/teams.py`, confirm analysis fns run for any pair.
2. Serialization: `src/serialize.py` + schema + `scripts/build_site_data.py`; produce
   real JSON for all teams locally.
3. Frontend skeleton: Vite app, data hooks, team page reading real JSON.
4. Landing + method pages; shareable takeaway; caveat badges.
5. Polish + responsive; Vitest smoke test.
6. GitHub Actions: auto-discover, cache, build, deploy to Pages.

## 10. Non-blocking / later
- Repo rename / product name (currently `antonelli-vs-russell`).
- Custom domain; social-preview/OG images per team for shareability.
- Optional Netlify/Vercel migration.
